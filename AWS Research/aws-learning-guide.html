Route 53 — Routing Policies
How AWS directs internet traffic to the right place
🤔 What is it? Think of Route 53 like a GPS for the internet. When someone types your website URL, Route 53 decides which server to send them to — and it can make smart decisions based on speed, location, health, and more.
🧠 Memory Trick
Route 53 = Internet Traffic Cop 🚦 — it stands at the intersection and points visitors to the right server based on rules you set.
01
➡️
Simple
One domain → One server. The most basic setup. Like a street sign pointing to one destination.
No health checksEasy setup
02
⚖️
Weighted
Split traffic by percentage. Send 90% to Server A and 10% to Server B — great for testing new features slowly.
A/B Testing0–255 weights
03
⚡
Latency
Automatically sends users to the fastest server for them. A user in Europe goes to Europe servers, not USA servers.
Fastest speedMulti-region
04
🔁
Failover
Primary server goes down? Route 53 automatically switches to a backup server. Like a spare tire for your website.
Needs health checksHigh availability
05
🗺️
Geolocation
Route users based on their country. Users in Egypt see Arabic content; users in Germany see German content.
By country/regionLocalization
06
📍
Geoproximity
Route by resource location, not user location. You can even "pull" traffic toward a region using bias values.
Traffic Flow neededBias control
07
🖥️
IP-Based
Route based on the user's IP address. Useful for directing corporate IPs to internal servers or specific ISPs.
By IP rangeISP control
08
🎲
Multi-Value
Returns up to 8 healthy servers at random. Clients pick one. Like a poor man's load balancer — simple and effective.
Up to 8 IPsNot a true LB
💡
Quick Decision Guide: Need backup? → Failover. Testing new features? → Weighted. Global users? → Latency. Different content per country? → Geolocation. Just starting out? → Simple.

EC2 — Pricing Models
5 ways to pay for your cloud servers
🤔 What is it? EC2 is AWS's virtual servers (computers in the cloud). Instead of buying one pricing model, AWS gives you 5 different ways to pay — each suited to different situations. Choosing the right one can save you up to 90% on your bill.
🧠 Memory Trick
Think of EC2 pricing like renting a car: 🚗 On-Demand = daily rental | 📅 Reserved = yearly lease | 🎲 Spot = standby seat (cheap but can be cancelled) | 💳 Savings Plan = loyalty card | 🏠 Dedicated = private chauffeur
01
🔓
On-Demand
Pay by the second, no commitment. Start and stop anytime. The most flexible but most expensive option.
No commitmentMost expensive
02
📅
Reserved Instances
Commit to 1 or 3 years, get up to 72% off. Like signing a long lease — you save big if you're sure you'll use it.
Up to 72% off1–3 yr lock-in
03
⚡
Spot Instances
Buy unused AWS capacity at huge discounts — up to 90% off! But AWS can reclaim it with 2-minutes notice. Great for batch jobs.
Up to 90% offCan be interrupted
04
💳
Savings Plans
Commit to a $/hour spend for 1–3 years. More flexible than Reserved — discounts apply across EC2, Lambda, and more services.
Up to 72% offMulti-service
05
🏠
Dedicated Hosts
A physical server just for you. Most expensive, but required for strict compliance or bring-your-own-license (BYOL) software.
Most expensiveFull isolation

Best Strategy: Start with On-Demand for 2–3 months to understand your usage. Then switch to Savings Plans or Reserved for steady workloads. Use Spot for background batch jobs to save maximum money.

S3 — Storage Classes
7 tiers of storage — from hot to frozen
🤔 What is it? Amazon S3 is cloud file storage. It has 7 storage classes — think of them like a temperature scale from 🔥 hot to ❄️ frozen. The "hotter" the data (accessed frequently), the more you pay. The "colder" (rarely accessed), the cheaper — but slower to retrieve.
🧠 Memory Trick — The Temperature Scale
🔥 Standard → 🌡️ Intelligent-Tiering → 🌤️ Standard-IA → ☁️ One Zone-IA → 🧊 Glacier Instant → ❄️ Glacier Flexible → 🥶 Deep Archive
Colder = Cheaper storage cost, but longer retrieval time

How these 3 services work together:
A user visits your website → Route 53 sends them to the nearest/fastest server → EC2 runs your application code → the app reads/writes files and data from S3. This trio forms the backbone of most AWS-powered applications!
